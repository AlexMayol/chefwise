# Database Layer

## SQLite Contract

- `client.ts` opens the Expo SQLite database and enables foreign keys.
- `schema.ts` is the source of truth for tables, indexes, and delete rules.
- `migrations.ts` applies schema changes and updates `PRAGMA user_version`.
- Keep migrations idempotent. Tests should be able to run migrations on an empty database repeatedly.
- Schema changes after V1 need a new migration step and `LATEST_SCHEMA_VERSION` bump; do not silently rewrite the v1 SQL and assume existing devices update.

## Repositories

- Repositories are thin SQL adapters. Keep workflow logic in `lib/domain/` or hooks.
- IDs are generated by `createId(prefix)` as `prefix-uuid-v4`.
- Use `insertRow` and `updateRow` for simple table operations unless custom SQL is clearer.
- Product prices intentionally expose create/read methods only. Do not add update/delete for historical price rows.

## Data Rules

- Products, markets, recipes, pantry, and shopping records use restrictive foreign keys where immutable or active records depend on them.
- Deleting a category sets `products.categoryId` to null.
- Pantry has one current item per product via `idx_pantry_items_product`.
- Store booleans as SQLite integers and map at repository boundaries.
- Full backup import replaces files on disk; an already-open SQLite handle may need an app restart or provider-level reopen to see restored data.
